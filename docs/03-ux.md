# Tier 3 - User Experience

Brings the UI to a polished, daily-use standard.

---

## 3.1 Per-member points on Leaderboard tab [done]

**Status:** Implemented. Member-level totals are computed by the API and rendered in the Leaderboard tab.

**Implemented approach:**
`POST /users/scores` returns tenant-scoped member score totals for the selected season. The dashboard passes those totals to the `Leaderboard` component.

The leaderboard also refreshes after point awards without requiring a full browser reload.

---

## 3.2 Leaderboard refreshes after awarding points [done]

**Current state:** After submitting the Award Points dialog, server revalidation refreshes the active-season dashboard and leaderboard data without a full browser reload.

**Deferred option:** True optimistic UI could still make scores move before the server confirms the award.

Potential future approach:

- Add a `onSuccess` callback to `AwardPointsDialog` that accepts the `{ targetHouseId, delta }` 
- In `DashboardShell`, maintain a local `Map<houseId, scoreOffset>` in state
- Apply the offset to the displayed score immediately, clearing it on next data fetch
- This is a cosmetic optimistic update; the real value comes back on revalidation

---

## 3.3 Activity feed cursor pagination API [done]

**Status:** The API supports bounded cursor pagination through `POST /transactions/recent`, and the dashboard uses it through the Activity load-more UI.

---

## 3.4 Activity feed load-more UI [done]

**Implemented approach:**

- The dashboard reads the first activity page on the server.
- `ActivityFeed` renders a "Load more" button when the API returns `nextCursor`.
- Clicking the button calls a cursor-aware Server Action and appends the returned items client-side.
- The button disappears when the feed is exhausted.
- Failed page loads show a safe retryable error without clearing already-loaded activity.

---

## 3.5 Admin: edit house color / description [done]

**Status:** Implemented in the admin management UI and API. The edit form preloads the selected house color and description so partial updates do not feel destructive.

---

## 3.6 Mobile header [done]

**Status:** Implemented. The house badge and award controls adapt for narrow screens, and the tab labels collapse to icon-focused navigation.

---

## 3.7 Admin/owner soft delete for point awards [done]

**Status:** Implemented. Owners and admins can soft delete point awards from Activity, live scoring excludes deleted transactions, and Manage Audit includes the deleted awards in the shared audit history.

---

## 3.8 Seasons UX pass [done]

**Status:** Implemented. The dashboard defaults to the active season and supports historical reporting without changing the current house-card standings.

**Implemented approach:**

- Overview includes a season selector for report widgets.
- Leaderboard member scores follow the selected season.
- Activity stays all-activity and includes season badges.
- Manage includes owner-only season start and rename controls.
- Overview can include a feature-flagged current-season status card showing the active season name, start date, and days remaining when an end date exists. Set `SHOW_SEASON_OVERVIEW_CARD="true"` on the web service to display it.

Future season UX work is tracked in [Seasons Architecture](./seasons-design.md), including activity filtering, season winner summaries, season comparison reports, and scheduled rollovers.

---

## 3.9 Manage Team layout and invite reporting [done]

**Status:** Implemented. The Manage Team section uses compact assignment, invite, and role-management cards instead of page-width controls.

**Implemented approach:**

- Assignment controls align as a compact labelled row on wider screens and stack on small screens.
- Invite token generation stays in a focused card with copy support.
- Owners can promote members to admins from an owner-only role-management card. Admins can see the card, but the control is disabled.
- The Team section includes invite activity reporting from the admin audit stream, showing recent token generation and use.

Future invite reporting could add raw active/expired token counts from `OrgInvite` if the current audit-backed report is not enough.

---

## 3.10 Owner/Admin Manage permissions [done]

**Status:** Implemented. Owners own org-level configuration; admins continue to operate member and points workflows.

**Implemented approach:**

- Manage shows Houses and Seasons to admins, but disables those tabs with an owner-only label.
- House creation/editing now requires `OWNER` at the API layer.
- Season start and rename now require `OWNER` at the API layer.
- Admin promotion now requires `OWNER` at the API layer and is audited.
- Admins retain access to Manage Overview, Team, Audit, member assignment, invites, point awards, and point deletion.

---

## 3.11 Manage Audit history [done]

**Status:** Implemented. Manage Audit is the single source of truth for administrative history instead of duplicating deleted point awards in a separate grid.

**Implemented approach:**

- The API exposes cursor-paged audit history with optional event-type filtering.
- The Audit tab uses full-history language, supports filter changes, and can load older audit events without a full page refresh.
- Deleted point awards appear as `POINT_DELETED` audit events alongside invites, house assignments, season starts, and role changes.
- The old recently deleted points grid was removed from the Audit tab to reduce reporting duplication.
